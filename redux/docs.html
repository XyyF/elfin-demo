独立的数据管理框架，不仅仅是在React中。极简的大小，压缩后不到1KB。

数据流程必定是：某个Actions → 通过Dispatch广播 → 在Reducer中返回新的State

概念介绍

Action

一个对象，可以理解为交互行为的抽象，包含着行为类型，以及携带的数据；整个数据更新的起始点就是Action的产生，决定了state如何改变；

  1. 在Action对象中，除了type是硬性要求字段，
{
    type: '', // 交互行为
    payload: {}, // 携带数据
    test: 1, // 自定义字段
  }

Action因为单个比较零散，可以通过工厂函数的模式创建Action；

reducer

控制着state的函数，在函数内部根据不同的action返回全新的state，如果在reducer中直接修改state，是无效的

  1. state的初始化
  1. 不同action对应的state状态
/**
   * state 数据，因为第一次初始化没有action注意初始化
   * action dispatch广播的行为
   */
  function reducer(state = 0, action) {};
  
  var store = Redux.createStore(reducer);

Store

作为面向开发者需求的接口，主要负责state全局状态的管理；

store.getState

获取当前状态下的state值，真实的state通过闭包保存，无法直接获取；

store.getState();

store.dispatch

在redux中广播Action行为，因为一个store可能对应着多个redux（参考combineReducers），这里就是通过广播触发reducer更新state；

  1. 触发更新state；
  1. 触发监听器执行；
store.dispatch({type: '', payload: 0});

store.subscribe

通过store进行订阅注册，当store变化时，触发回调，多数用来进行UI同步；

  1. 内部维护一个监听器队列；
  1. 返回一个注销监听的方法；
const unsubscribe = store.subscribe(function() {});

store.replaceReducer

替换当前的reducer，因为在reducer中是返回全新的state，所以替换reducer十分方便；

store.replaceReducer(newReducer);

核心API

Rudex.createStore

接受一个reducer函数，返回一个store的方法；在创建store的同时，会初始化state，对应的Action为 ActionTyps.INIT（有加盐，防止重复）；

const randomString = () =>
    Math.random().toString(36).substring(7).split('').join('.')
  
  const ActionTypes = {
    INIT: `@@redux/INIT${/* #__PURE__ */ randomString()}`,
  }

Redux.bindActionCreators

将Action函数和store.dispatch进行绑定后返回新的广播函数，好处是广播时无需关注store在哪儿

function incrementAction() {
    return { type: 'INCREMENT' };
  };
  const increment = Redux.bindActionCreators(incrementAction, store.dispatch);

Redux.combineReducers

在大型应用的redux应用中，会将state拆分为不同namespace作用域下的状态，但应用只能有一个store，即createStore方法只能调用一次，所以需要将多个不同的reducer为一个store

function todoer(state = []) { return state };
  
  // { counter: 0, todoer: [] }
  Redux.createStore(
    Redux.combineReducers({counter, todoer})
  );
  
  store.replaceReducer(
    Redux.combineReducers({counter, todoer})
  );

Rudex.applyMiddleware

Rudex中的中间件概念，通过改造 dispatch，让我们能在dispatch前后进行一些自定义的操作；（如日志）

function applyMiddleware(...middlewares) {
    return function(createStore) {
      return function(reducer, payloadedState) {
        const store = createStore(reducer, preloadedState);
        // 在applyMiddleware中不允许通过store调用dispatch
        let dispatch: Dispatch = () => {
          throw new Error(
            'Dispatching while constructing your middleware is not allowed. ' +
              'Other middleware would not be applied to this dispatch.'
          )
        };
  
        // 第一函数参数，用以获取当前state状态
        const middlewareAPI: MiddlewareAPI = {
          getState: store.getState,
          dispatch: (action, ...args) => dispatch(action, ...args),
        };
        const chain = middlewares.map(middleware => middleware(middlewareAPI));
        // 第二函数参数，同时返回改造后的dispatch
        dispatch = compose(...chain)(store.dispatch);
        // ...
        return {
          ...store,
          dispath, // 改造后的 dispatch
        }
      }
    }
  }

中间件研发

// 研发者角度
  const logger1 = store => next => action => {
    console.group(action.type);
    console.info('logger1 dispatching', action);
    next(action);
    console.log('logger1 new state', store.getState());
    console.groupEnd(action.type);
  }
  
  // 使用者角度
  var store = Redux.createStore(
    counter,
    Redux.applyMiddleware(logger1),
  );

实例分析

applyMiddleware(log1, log2, log3, log4, ...);
  
  // 经过该步骤后，chain的值为 (next) => {....}
  const chain = middlewares.map(middleware => middleware(middlewareAPI));
  
  // 经过该步骤后
  compose(...chain);
  // ==>
  function() {
    return function() {
      return function() {
        // ...
        return f1(f2());
      }(fnn-1());
    }(fnn());
  };
  
  // 经过该步骤后
  // 1. fnn最先执行，它的参数是 dispatch！
  // 2. fnn-1之后执行，它们的参数是 (action) => {} ，即是fnn的执行返回值！
  // 3. fn1是最后执行，也就是最终的返回值是fn1！
  compose(...chain)(store.dispatch);
  
  // 最终执行dispatch
  // 1. fn1最先执行，接受参数action
  // 2. fn1在执行next前，state是旧数据
  // 3. fn1执行next时，直接执行fn2，如此如果没有中断，那么就是 fn1 -> fn2 -> fn3 -> ... fnn -> dispatch -> 更新数据
  // 4. 执行真正的dispatch后，将从fnn优先执行，此时已经是新数据
  // 总结: fn1(next前) -> fn2(next前) -> ... -> fn(next前) -> dispatch -> fn(next后) -> ... -> fn2(next后) -> fn1(next后)
  store.dispatch(action);

Rudex.compose

工具类函数，接受一串函数，将他们以嵌套形式返回，最终是返回一个全新的函数，将会倒叙执行所有的函数；

function compose(...funcs: Function[]) {
    if (funcs.length === 0) {
      // infer the argument type so it is usable in inference down the line
      return <T>(arg: T) => arg
    }
  
    if (funcs.length === 1) {
      return funcs[0]
    }
  
    return funcs.reduce(
      (a, b) =>
        (...args: any) =>
          a(b(...args))
    )
  }

案例分析

compose(fn1, fn2, fn3, fn4);
  
  // ==>
  function() {
    return f1(f2());
  };
  // ==>
  function() {
    return function() {
      return f1(f2());
    }(f3());
  };
  // ==>
  function() {
    return function() {
      function() {

      }(f3());
    }(f4());
  };
